* What is an ORM?
An *ORM* (Object-Relational Mapper) is a tool that lets your application work with a database using objects and classes instead of writing raw SQL queries all the time.

Think of it like a translator between:
- C# objects/classes ↔ Database tables/rows

** Without ORM
You manually write SQL:
#+begin_src sql
  SELECT * FROM Users WHERE Id = 1
#+end_src

Then manually map the result to a C# object.

** With ORM
You write C#:
#+begin_src csharp
  var user = context.Users.Find(1);
#+end_src

The ORM automatically:
- Generates SQL
- Executes it
- Converts database rows into C# objects

* What is Entity Framework?
Entity Framework Core (EF Core) is Microsoft's official ORM for .NET and ASP.NET Core applications.

It allows developers to:
- Interact with databases using C#
- Avoid most manual SQL writing
- Manage database schema using code
- Work with strongly typed models

Official documentation: [[https://learn.microsoft.com/en-us/ef/core/][Entity Framework Core Documentation]]

* Why is Entity Framework Used?
EF Core is mainly used to make database operations:
- Easier
- Faster to develop
- Less error-prone
- More maintainable

* Real-World Example
Imagine you have a Users table in SQL Server.

| Id | Name | Email         |
|----+------+---------------|
|  1 | John | john@test.com |

- In EF Core You Create a Class
#+begin_src csharp
  public class User
  {
      public int Id { get; set; }
      public string Name { get; set; }
      public string Email { get; set; }
  }
#+end_src

EF Core maps this class to the database table automatically.

* Core Concepts of EF Core

** 1. Entity
An Entity is a C# class mapped to a database table.

#+begin_src csharp
  public class Product
  {
      public int Id { get; set; }
      public string Name { get; set; }
  }
#+end_src

- Class = Table
- Properties = Columns

** 2. DbContext
DbContext is the main class EF uses to communicate with the database.

#+begin_src csharp
  public class AppDbContext : DbContext
  {
      public DbSet<Product> Products { get; set; }
  }
#+end_src

Think of it as:
- Database session
- Connection manager
- Query executor

** 3. DbSet
~DbSet<T>~ represents a table.

#+begin_src csharp
  context.Products
#+end_src

This means:
- Access the Products table
- Query/update/delete data

* CRUD Operations in EF Core
CRUD = Create, Read, Update, Delete

** Create
#+begin_src csharp
  var product = new Product
  {
      Name = "Laptop"
  };

  context.Products.Add(product);
  context.SaveChanges();
#+end_src

Generated SQL:
#+begin_src sql
  INSERT INTO Products ...
#+end_src

** Read
#+begin_src csharp
  var products = context.Products.ToList();
#+end_src

Generated SQL:
#+begin_src sql
  SELECT * FROM Products
#+end_src

** Update
#+begin_src csharp
  var product = context.Products.Find(1);

  product.Name = "Gaming Laptop";

  context.SaveChanges();
#+end_src

** Delete
#+begin_src csharp
  var product = context.Products.Find(1);

  context.Products.Remove(product);

  context.SaveChanges();
#+end_src

* How EF Core Works Internally

When you write:
#+begin_src csharp
  var users = context.Users.Where(u => u.Name == "John").ToList();
#+end_src

EF Core:
1. Converts LINQ into SQL
2. Sends SQL to database
3. Gets results
4. Converts rows into objects

* What is LINQ?
*LINQ* = /Language Integrated Query/. It allows querying data using C# syntax.

** Example
#+begin_src csharp
  var users = context.Users
      .Where(u => u.Age > 18)
      .OrderBy(u => u.Name)
      .ToList();
#+end_src

Instead of SQL:
#+begin_src sql
  SELECT * FROM Users
  WHERE Age > 18
  ORDER BY Name
#+end_src

* Benefits of Entity Framework Core

** 1. Faster Development
Less SQL writing.

** 2. Strongly Typed
Compile-time checking reduces errors.

** 3. Database Independence
Can work with:
- SQL Server
- PostgreSQL
- MySQL
- SQLite

** 4. Migrations Support
You can create/update database schema using code.

Example:
#+begin_example
  dotnet ef migrations add InitialCreate
  dotnet ef database update
#+end_example

** 5. LINQ Support
Write queries in C# instead of raw SQL.

** 6. Change Tracking
EF automatically tracks modified objects.

#+begin_src csharp
  product.Name = "New Name";
  context.SaveChanges();
#+end_src

No manual UPDATE query needed.

* EF Core Architecture (Simple View)

#+begin_example
  ASP.NET Core App
         ↓
  Entity Framework Core
         ↓
  Database Provider
         ↓
  SQL Server / PostgreSQL / MySQL
#+end_example

* Database First vs Code First

** 1. Code First
- You create C# classes first.
- EF creates database tables.
- *Best for:* New projects

** 2. Database First
- Database already exists.
- EF generates classes from database.
- *Best for:* Existing databases

* When Should You Use EF Core?
EF Core is great when:
- Building business applications
- Rapid development matters
- You want maintainable code
- You use ASP.NET Core APIs
- You prefer C# over SQL-heavy code

* When EF Core May Not Be Ideal
Sometimes raw SQL or micro ORMs are better:
- Very high-performance systems
- Extremely complex SQL
- Heavy reporting systems

Example micro ORM: **Dapper**

* Small Complete Example

** Model
#+begin_src csharp
  public class Student
  {
      public int Id { get; set; }
      public string Name { get; set; }
  }
#+end_src

** DbContext
#+begin_src csharp
  public class SchoolContext : DbContext
  {
      public DbSet<Student> Students { get; set; }

      public SchoolContext(DbContextOptions<SchoolContext> options)
          : base(options)
      {
      }
  }
#+end_src

** Controller Usage
#+begin_src csharp
  public class StudentsController : Controller
  {
      private readonly SchoolContext _context;

      public StudentsController(SchoolContext context)
      {
          _context = context;
      }

      public IActionResult Index()
      {
          var students = _context.Students.ToList();

          return View(students);
      }
  }
#+end_src
